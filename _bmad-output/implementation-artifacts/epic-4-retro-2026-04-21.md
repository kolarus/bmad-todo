## What Went Well

### 1. BMAD Artifact Chain Eliminated Mid-Sprint Ambiguity
The PRD → Architecture → Epics → Stories chain meant every story carried its own full context: requirements mapping, UX spec references, architecture decisions. Stories were self-contained and executable. This was called out by both the technical team (Charlie) and the project lead (Nearformer) as the single most impactful process choice.

### 2. Dev Notes as Working Code Examples — A Force Multiplier
All 9 Epic 4 stories (and stories across all epics) included complete, working code examples in their Dev Notes sections. For example:
- Story 4.1: Complete `playwright.config.ts`, `clearAllTodos` helper implementation
- Story 4.5: `page.route()` error simulation pattern
- Story 4.6: `page.route()` with 500ms artificial latency for optimistic UI timing verification
- Story 4.7: `vitest.config.ts` coverage configuration block

This reduced mid-story ambiguity to near zero and likely accounts for the clean test suite (0 failures).

### 3. Test Isolation via API-Level Setup/Teardown — Zero Flaky Tests
The `clearAllTodos` API call in `beforeEach` for every E2E test, combined with proper `waitForResponse` patterns, produced 12 E2E tests with no flakiness. 110 total tests — 0 failures across all terminal runs through the project lifecycle.

### 4. Scope Discipline
The project scoped a todo app, built a todo app, and shipped a todo app. No scope creep. Every epic delivered exactly what it defined. This is rarer than it sounds.

### 5. WCAG 2.1 AA Compliance Built In (Not Bolted On)
Accessibility was delivered as a full Epic 3 deliverable, not an afterthought. Keyboard navigation, focus indicators, ARIA labels, semantic HTML, and 4.5:1 contrast ratios were acceptance criteria — not "nice to haves."

## What Was Hard / Challenges

### 1. Story 4.1 Cross-Epic Infrastructure Reach
The E2E infrastructure setup story (4.1) required adding `DELETE /api/todos` (bulk delete) to the backend — a cross-cutting concern from an Epic 4 story reaching into Epic 2 territory. This was an undocumented cross-epic dependency that worked cleanly but wasn't planned explicitly.

### 2. `data-testid` Dependency Was Implicit, Not Explicit
Stories 4.2–4.6 all relied on `data-testid` attributes (`todo-input`, `todo-item`, `todo-checkbox`, `error-state`, etc.) being present on frontend elements. These were set in Epic 2/3 stories. No story or acceptance criterion in Epics 2/3 explicitly said "add `data-testid` for E2E compatibility." It worked because the implementation was correct — but it was an undocumented assumption, not a designed contract.

### 3. Backend Coverage 69.23% — A Pass With an Asterisk
The 70% coverage target was met at the overall level, but only because `server.ts` (44%) and `env.ts` (50%) were excluded from the threshold justification. The service/route/validation layers are properly covered (85–100%). However, the per-file coverage thresholds were not explicitly set in `vitest.config.ts` — the justification was narrative, not enforced by tooling.

### 4. QA Report Stories Are a Different Work Type
Stories 4.7, 4.8, and 4.9 (coverage report, security review, QA summary) are "analysis and documentation" stories, not "code" stories. They required running tools (`npm audit`, Lighthouse, coverage commands) and writing structured analysis — not implementing features. Future teams may underestimate or overestimate these stories if they're mixed with implementation stories without a type distinction.

### 5. Dependency Chain Risk
Each epic built on the last. Epic 4's E2E tests were full-stack integration tests for Epics 1–3. If any prior epic had defects, they would surface in Epic 4's tests — with potentially cascading rework. This worked because the prior epics were sound, but the risk was real and unmitigated.

---

## Key Insights

1. **The BMAD planning artifacts are the product of the planning epics** — they are not overhead. They are the leverage that made 110 tests pass on first delivery.
2. **Dev Notes in story files should be treated as a first-class deliverable** of story planning, not optional commentary.
3. **E2E tests own the full-stack regression story.** Unit and component tests don't need to duplicate E2E scenarios — they test units. The E2E suite is the integration layer.
4. **`clearAllTodos` via API (not UI) before each E2E test is the canonical isolation pattern.** UI-level cleanup introduces flakiness and coupling.
5. **Cross-epic `data-testid` contracts need to be explicit.** They worked by luck of good implementation, not by design. Future projects should formalize this.
